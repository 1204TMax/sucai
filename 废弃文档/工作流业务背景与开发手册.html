工作流模块开发手册

版本: v1.0 分类: 业务+技术规格书 最后更新: 2026-04-26

1. 业务背景与价值

在量化投放领域，素材生产正从“单点创作”转向“工业化裂变”。

核心痛点

传统 AI 生成质量不稳定。如果等到 100 张全生完再挑，反馈周期太长；如果全自动不审核，则会浪费算力和产生大量垃圾素材。

核心愿景：构建一个“半自动、可干预、流式消费”的生产流水线。让人类负责审美决策，且与 AI 生产并行。

2. 核心架构模型

实体关系 (ER)

  • Workflow (工作流)：生产逻辑的“剧本/模板”。
  • Task (任务)：运行一次工作流产生的具体“剧组”。
  • Item (产出物)：任务在运行中产出的最小单元。
  • Raw Material (原料)：Item 被确认后的持久化形态。

核心组件分工

  • 广场：分发中心，解决“我该用哪个工作流”的问题。
  • 编辑器：设计中心，解决“生产线怎么搭建”的问题。
  • 审查台：决策中心，解决“哪些产物可以用”的问题。

3. 深度逻辑：边生边审

这是本系统最复杂也最重要的逻辑，开发者必须深刻理解：

// 逻辑示意：流式消费
On (Item_Generated) {
    PushTo (Review_Queue);
    If (User_Passes_Item) {
        Trigger_Next_Node(Item); // 立即触发下游，不要等其他 Item
    }
}

为什么必须是流式？

如果是批处理（全部生完再审），用户将面临极长的等待时间。流式处理让生产线“动起来”，极大提升了人机协作的效率。

4. 各页面开发要点

4.1 通用生成 (重构)

将 Tab 升级为“文案”，并引入“广场”的三段式布局。支持参考图输入。

4.2 工作流编辑器 (新增)

需要实现变量引用语法，例如 {{Node1.output_text}}。每个节点点击后滑出参数抽屉。

4.3 我的任务 (重构)

任务卡片需展示聚合进度，例如：Total 100 | Generated 20 | Pending 10 | Passed 5。
