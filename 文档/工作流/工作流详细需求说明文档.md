# 工作流系统总需求文档

> 本文档为 AI 创作模块中「工作流」能力的详细需求说明文档，面向产品、设计、研发、测试和后续 AI Coding 执行使用。
>
> **SSOT 原则**：本文档是工作流功能的唯一事实来源。后续不再创建 `_v2`、`_final`、`最新版` 等副本文档。工作流相关需求变化必须直接更新本文档。
>
> **当前版本口径**：本文档已吸收 2026年4月27日工作流确认会议结论，并补充 2026年4月28日「我的任务」文案任务卡片展示规则。若早期文档中存在“工作流广场放在通用生成页内”“物料级流式审查”“任务卡片展示 4 个 loading 位”“多条文案合并成一段编号文本”等旧口径，以本文档为准。

## 一、项目概述

### 1.1 项目背景

- 当前 AI 创作模块已经具备文案、图像、视频等单点生成能力，但业务实际生产经常需要多步骤串联，例如“爆款素材拆解 → 文案衍生 → 图片生成 → 人工审查 → 视频合成 → 输出结果”。
- 仅依赖通用生成会导致用户反复手动复制提示词、上传素材、筛选结果和发起下一步任务，生产链路长、重复操作多、过程不可追踪。
- 工作流能力的目标不是把所有生成过程变成完全黑盒，而是提供一套可配置、可复用、可审查、可追踪的半自动生产流水线。
- 工作流系统需要同时服务两类用户：
  - 工作流创建者：懂业务、会配置流程，希望沉淀生产方法。
  - 工作流使用者：只想快速使用成熟流程，不需要理解全部节点细节。
- 因此，工作流系统既要支持较复杂的画布式搭建，也要在使用和任务查看阶段尽量降低认知成本。

### 1.2 项目目标

构建一套围绕 AIGC 生产链路的工作流系统，让业务、产品、设计、研发能够基于同一套定义推进页面、交互、状态和接口设计：

1. 支持用户创建、编辑、发布和使用工作流。
2. 支持工作流按主产物类型归类为文案、图像、视频。
3. 支持工作流绑定项目，用于读取项目素材、历史文案、产品信息等上下文。
4. 支持工作流通过节点画布编排多步骤生产链路。
5. 支持人工审查节点作为工作流中的 Check Point。
6. 支持工作流任务进入“我的任务”，并按最终产物类型融合展示。
7. 支持工作流运行中展示简化流程状态，让用户知道当前是否需要自己处理。
8. 支持工作流结果回流到结果页、任务中心和后续素材/原料管理链路。
9. 支持工作流广场作为独立页面，用于发现、收藏和使用更多工作流。
10. 统一页面范围、字段口径、状态流转、按钮动作、跳转关系和设计约束。

### 1.3 核心设计原则

| 原则 | 说明 |
|---|---|
| 使用入口轻量 | 通用生成页只展示收藏和我的工作流，不承载完整广场，避免干扰单点创作。 |
| 广场独立承载 | 工作流广场作为独立菜单页，负责发现、筛选、收藏和使用公开工作流。 |
| 创建与使用分离 | 创建者在编辑器里理解节点细节；使用者在运行时只填写必要输入并查看结果。 |
| 单步多步兼容 | 工作流通常用于多步骤生产链路，但也允许封装单步最佳实践、提示词模板和固定业务场景。 |
| 运行状态聚合 | 我的任务不暴露完整复杂节点图，只展示开始、当前处理点、待审查、结束等简化状态。 |
| 审查节点可多个 | 工作流中可配置多个审查节点，系统不限制只能有一个人工审查。 |
| 审查按节点聚合 | 审查不按每个物料碎片化打扰用户，而是按审查节点或批次聚合，让用户一次性处理。 |
| 结果按类型归类 | 工作流任务不单独开 Tab，按最终产物类型进入文案、图像、视频任务流。 |
| 文案列表展示 | 文案结果必须使用垂直列表，一行一条，不使用网格卡片，不使用聊天窗口样式。 |
| 页面风格统一 | 所有页面保持统一、稳定、紧凑的系统视觉风格。 |
| 原型优先稳定 | 原型阶段优先保证可预览、可演示、可稳定传递需求，不以复杂技术形态为目标。 |
| 直接打开可看 | 当前原型应尽量降低预览门槛，确保评审和确认成本可控。 |

## 二、系统架构

### 2.1 页面层级结构

工作流系统归属于「AI 创作」模块，由固定菜单页和动作进入页共同组成：

```text
AI 创作（固定菜单）
├── 通用生成                  ← 单点创作入口 + 工作流快捷入口
│   ├── 新建工作流弹窗         ← 采集工作流基础信息
│   └── 工作流编辑器           ← 创建后进入画布编辑
├── 我的任务                  ← 图像 / 视频 / 文案任务融合展示
│   ├── 工作流运行态卡片       ← 展示当前流程状态和审查入口
│   ├── 审查台                ← 处理工作流人工审查任务
│   └── 全部结果页             ← 查看任务完整产物
├── 工作流广场                ← 独立广场，用于发现、筛选、收藏、使用工作流
├── 工作流编辑器              ← 工作流搭建与编辑页，通常由新建/编辑动作进入
├── 原料库                    ← 后续增强页，承接多模态原料与溯源，不作为本轮强依赖
└── 项目管理                  ← 项目数据来源和上下文管理
```

### 2.2 文件与菜单约定

| 页面 | 文件 | 是否固定菜单 | 说明 |
|---|---|---|---|
| 通用生成 | `AI 创作/通用生成.html` | 是 | 单点创作入口，展示收藏和我的工作流快捷入口 |
| 我的任务 | `AI 创作/我的任务.html` | 是 | 统一任务中心，工作流任务按最终产物类型融合展示 |
| 工作流广场 | `AI 创作/工作流广场.html` | 是 | 独立广场，承接工作流发现与收藏 |
| 工作流编辑器 | `AI 创作/工作流编辑器.html` | 可固定，也可动作进入 | 工作流画布搭建页 |
| 审查台 | `AI 创作/我的任务.html?view=review...` 或独立承接页 | 否 | 从任务卡片进入，处理审查任务 |
| 全部结果页 | `AI 创作/全部结果-*.html` | 否 | 查看普通任务完整结果 |
| 工作流全部结果页 | `AI 创作/工作流全部结果页.html` | 否 | 查看工作流任务完整结果 |
| 原料库 | `AI 创作/原料库.html` | 可固定 | 后续承接原料与溯源，本轮不作为工作流 MVP 必须依赖 |

### 2.3 核心对象模型

```text
Workflow（工作流定义）
├── 基础信息
├── 项目归属
├── 主产物类型
├── 封面信息
├── 权限与可见性
├── 节点定义
├── 连线定义
├── 运行输入表单定义
├── 发布状态
└── 创建/更新时间

WorkflowNode（工作流节点）
├── 节点 ID
├── 节点类型
├── 节点名称
├── 输入桩/输出桩
├── 配置参数
├── 上游变量引用
├── 测试状态
└── 运行状态

WorkflowRun（工作流运行任务）
├── 任务 ID
├── 工作流 ID
├── 所属项目
├── 主产物类型
├── 用户运行输入
├── 当前运行状态
├── 节点运行快照
├── 审查任务聚合
├── 结果统计
└── 创建/完成时间

ReviewTask（审查任务）
├── 审查任务 ID
├── 所属 WorkflowRun
├── 所属审查节点
├── 待审内容集合
├── 已通过内容集合
├── 已拒绝内容集合
├── 未运行到数量
└── 审查状态

WorkflowResult（工作流产物）
├── 产物 ID
├── 产物类型
├── 文件/文本内容
├── 来源工作流 ID
├── 来源任务 ID
├── 来源节点 ID
├── 审查状态
├── 入库状态
└── 生成时间
```

### 2.4 关键数据流

```text
通用生成页 ──新建工作流──→ 新建工作流弹窗 ──创建──→ 工作流编辑器
工作流编辑器 ──保存草稿/发布──→ 工作流定义

工作流广场 ──收藏──→ 通用生成页「收藏」
工作流广场 ──使用──→ 工作流运行输入表单 ──提交──→ 我的任务

通用生成页「我的/收藏」──使用──→ 工作流运行输入表单 ──提交──→ 我的任务

我的任务 ──工作流运行中──→ 工作流运行态卡片
我的任务 ──需要审查──→ 审查台 ──通过/拒绝──→ 工作流继续运行
我的任务 ──运行完成──→ 最终结果卡片 ──查看全部──→ 全部结果页

工作流结果 ──附带 WF-ID / TSK-ID / Node-ID──→ 后续原料库、素材库、物料库或结果页追溯
```

核心约束：

- 通用生成页负责快捷使用，不负责完整广场浏览。
- 工作流广场负责发现和收藏，不负责复杂编辑。
- 工作流编辑器负责搭建，不负责任务流结果展示。
- 我的任务负责运行状态和结果，不负责修改工作流结构。
- 审查台负责人工处理，不负责展示完整编辑画布。

## 三、页面详细设计

### 3.1 通用生成页

文件：`AI 创作/通用生成.html`  
入口：左侧导航「通用生成」；首页快捷入口  
职责：承接文案、图像、视频的单点创作，同时展示与当前创作类型相关的工作流快捷入口。

#### 3.1.1 页面结构

通用生成页由两部分组成：

| 区域 | 说明 |
|---|---|
| 单点创作区 | 原有文案、图像、视频生成能力，由顶部全局类型切换控制 |
| 工作流快捷区 | 展示收藏工作流和我创建的工作流，跟随顶部全局类型切换 |

工作流快捷区不再包含内部「文案 / 图像 / 视频」二级切换，避免和顶部全局切换产生割裂。

页面上半区的单点创作区域应整体处于首屏居中偏上的位置；标题、类型切换和输入框属于同一个首屏创作区，应作为整体保持统一的纵向层级。

工作流区域顶部应展示跟随全局类型切换的动态标题，例如「文案工作流」「图像工作流」「视频工作流」。

「新建工作流」应作为当前类型工作流区的重要入口，保证用户在浏览工作流时能够快速发起创建。若新用户「我的」列表为空，仍需保留「查看工作流广场」和「新建工作流」引导，帮助用户理解下一步可做什么。

当当前创作类型下收藏工作流数量为 0 时，不展示「收藏」标题和收藏区域，避免空区域占用页面。

#### 3.1.2 数据字段

| 字段 | 类型 | 说明 |
|---|---|---|
| 当前创作类型 | enum | 文案 / 图像 / 视频，由页面顶部全局切换控制 |
| 收藏工作流列表 | Workflow[] | 当前用户收藏的工作流，按当前创作类型过滤 |
| 我的工作流列表 | Workflow[] | 当前用户创建的工作流，按当前创作类型过滤 |
| 工作流名称 | string | 卡片主标题 |
| 工作流描述 | string | 卡片说明 |
| 主产物类型 | enum | 文案 / 图像 / 视频 |
| 所属项目 | string | 工作流绑定项目或适用项目 |
| 创建者 | string | 工作流创建人 |
| 创建时间 | datetime | 工作流创建时间 |
| 封面 | object | 自动文字封面或用户上传封面 |
| 收藏状态 | boolean | 是否已收藏 |
| 使用次数 | number | 用于衡量工作流被使用情况 |
| 收藏次数 | number | 用于衡量工作流被收藏情况 |
| 可见性 | enum | 私有 / 团队公开 / 广场公开 |

#### 3.1.3 功能点

- 支持在同一页面完成单点文案、图像、视频生成。
- 工作流快捷区跟随顶部全局类型切换，只展示当前类型可用的工作流。
- 工作流快捷区只展示「收藏」和「我的」，不展示完整「广场」。
- 「收藏」用于快速访问从工作流广场收藏的工作流。
- 「我的」用于快速访问当前用户创建的工作流。
- 当用户想查看更多工作流时，通过「查看工作流广场」跳转到独立工作流广场页。
- 当用户点击「新建工作流」时，系统会检查是否存在未完成的工作流草稿：
  - 如果存在，提示“检测到有未完成的工作流，是否进入上次编辑的工作流？”。用户可选择「进入上次编辑」（继承草稿进入编辑器）或「新建工作流」（清空草稿并正常打开弹窗）。
  - 如果不存在，直接打开标题为「新建工作流」的弹窗。
- 工作流卡片支持整卡直接使用当前工作流；卡片同时保留收藏入口和更多操作入口。
- 自己创建的工作流应支持编辑、创建副本和删除；非自己创建的工作流仅支持创建副本。
- 工作流卡片应统一标题、说明和创建信息的基础信息结构。
- 通用生成页工作流卡片不展示类型、收藏数、使用次数等发现型统计信息，以避免干扰快捷使用。
- 收藏 / 取消收藏应真实改变当前页面列表：收藏后进入「收藏」区域，取消收藏后从「收藏」区域移除；原型演示可采用运行时状态，刷新后恢复默认数据。
- 删除应先经过二次确认，确认文案为「是否删除该工作流？正在进行的任务不会受到影响。」；确认后再从当前页面列表移除。原型演示不要求持久化删除，刷新后恢复默认数据。
- 单点创作区用于解决用户即时的单点生产诉求，可以支持多模态输入，但不应做成通用大模型聊天页的重复产品。
- 工作流快捷区用于让用户使用已经封装好的业务生产方法。工作流可以是多步链路，也可以是单步模板化工具，关键区别在于它是可复用、可管理、可沉淀的生产资产。

#### 3.1.4 新建工作流弹窗

新建工作流弹窗（标题为「新建工作流」）用于采集创建工作流所需的最小基础信息。创建成功后进入工作流编辑器。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 名称 | string | 是 | 工作流名称，用于卡片、任务和编辑器标题 |
| 主产物类型 | enum | 是 | 文案 / 图像 / 视频，只能选择一个 |
| 描述 | textarea | 否 | 说明该工作流适用场景和能力 |
| 可见性 | enum | 是 | 仅自己可见 / 团队公开 |
| 封面 | object | 否 | 默认自动生成，也支持上传图片 |

封面规则：

- 封面不是第一必填项，不应让用户误以为创建前必须上传图片。
- 默认封面采用“名称关键字 + 色块”的方式生成，参考钉钉群封面逻辑。
- 系统从工作流名称中提取 4-6 个字作为封面文字。
- 默认色块可自动分配，也允许用户修改颜色。
- 用户可选择上传自定义图片作为封面；上传后会替换默认色块并展示图片预览。
- 上传封面后，支持一键「删除」，删除后自动恢复为默认提取的文字色块封面。

#### 3.1.5 操作按钮

| 按钮 | 说明 | 结果 |
|---|---|---|
| 文案 / 图像 / 视频 | 切换当前创作类型 | 单点创作区和工作流快捷区同步切换 |
| 新建工作流 | 创建新的工作流 | 打开新建工作流弹窗 |
| 查看工作流广场 | 查看更多公开工作流 | 打开 `工作流广场.html` |
| 点击卡片 | 直接使用当前工作流 | 打开运行输入表单或创建工作流任务 |
| 编辑 | 编辑自己创建的工作流 | 打开 `工作流编辑器.html` |
| 创建副本 | 基于当前工作流创建副本 | 生成可编辑副本 |
| 删除 | 删除自己创建的工作流 | 二次确认后移出当前列表 |
| 收藏 / 取消收藏 | 更新收藏状态 | 当前工作流加入或移出收藏列表 |
| 创建并进入画布 | 提交新建弹窗 | 创建工作流草稿并打开编辑器 |
| 取消 | 关闭新建弹窗 | 不创建工作流 |

### 3.2 工作流广场页

文件：`AI 创作/工作流广场.html`  
入口：左侧导航「工作流广场」；通用生成页「去工作流广场」  
职责：展示公开或可见范围内的工作流，支持筛选、搜索、收藏、使用和查看。

#### 3.2.1 页面结构

| 区域 | 说明 |
|---|---|
| 主产物类型切换区 | 页面顶部展示文案 / 图像 / 视频三大类切换，用于决定当前浏览的工作流类型 |
| 搜索筛选区 | 按关键词、创建者、创建时间和来源条件筛选工作流 |
| 排序区 | 支持按使用次数或按创建时间切换列表顺序 |
| 工作流列表区 | 以卡片展示工作流信息 |
| 空状态 | 无匹配工作流时展示空状态和引导 |

工作流广场是“大卖场”定位，不和通用生成页混在一起。

展示原则：

- 顶部工具区应保持紧凑、清晰，避免过强的表单感，优先让用户聚焦于工作流列表本身。
- 工作流广场卡片与通用生成页工作流卡片保持一致的基础信息结构，降低跨页面认知成本；广场卡片额外补充类型、收藏数和使用次数等发现型信息。

#### 3.2.2 数据字段

| 字段 | 类型 | 说明 |
|---|---|---|
| 关键词 | string | 搜索工作流名称、描述、创建者 |
| 主产物类型 | enum | 文案 / 图像 / 视频 |

#### 3.3.3 节点类型规划

| 节点类型 | 说明 | v1.0 要求 |
|---|---|---|
| 开始节点 | 定义运行时用户需要填写的表单输入 | 必须支持 |
| 结束节点 | 定义最终输出类型和结果归属 | 必须支持 |
| 文本处理节点 | 文案生成、改写、总结、提取卖点等 | 必须支持 |
| 图像处理节点 | 图片生成、图片改写、图生图等 | 必须支持 |
| 视频处理节点 | 视频生成、视频合成、视频改写等 | 必须支持 |
| 素材拆解节点 | 输入爆款视频，拆出音轨、片段、字幕和结构 | v1.0 重点支持 |
| 素材衍生节点 | 基于拆解结构或输入素材做裂变 | v1.0 重点支持 |
| 自动混剪节点 | 将通过审查的素材按规则组合为成片 | v1.0 重点支持 |
| 人工审查节点 | 暂停自动流转，由人工通过/拒绝 | 必须支持 |

通用能力内聚原则：

- 去水印、等比例裁剪、视频超分、格式转换等通用能力不作为独立节点优先暴露。
- 这类能力应作为图像处理节点或视频处理节点中的配置项存在。
- 避免让用户为了高频小动作连接大量低价值节点。

工作流步数原则：

- 工作流不强制必须多步。若一个节点已经足以承载稳定业务场景，也可以形成单步工作流。
- 单步工作流适合沉淀提示词模板、固定输入表单、项目上下文和默认参数；当前生成节点可通过 Prompt + 变量引用 + 参考内容的方式封装单步能力。
- 多步工作流适合沉淀需要拆解、衍生、审查、合成等环节的复杂生产链路。
- 无论单步还是多步，都必须通过开始节点定义运行输入，并通过结束节点定义最终产物类型。

#### 3.3.4 画布交互

- 支持从左侧节点库拖拽节点到画布。
- 支持节点在画布上自由拖动。
- 支持画布左右、上下平移。
- 支持画布缩放，缩放用于查看全貌和快速定位。
- 支持从输出桩拖拽到输入桩建立连线。
- 支持拖动已有连线端点重新连接。
- 支持双击连线删除。
- 支持删除节点，并同步删除关联连线。
- 支持点击节点查看和编辑节点配置。
- 支持开始节点和结束节点默认存在。
- 开始节点和结束节点原则上各只能保留一个。
- 开始节点和结束节点属于固定节点：初始化即存在，不支持删除。
- 工作流基础信息应支持编辑名称、描述、封面和可见性权限。

#### 3.3.5 开始节点规则

开始节点定义“用户使用这个工作流时需要填什么”。

| 字段 | 类型 | 说明 |
|---|---|---|
| 输入项名称 | string | 如“裂变方向”“参考图片”“视频素材”“目标数量” |
| 输入项类型 | enum | 文本 / 图片 / 视频 / 音频 / 数量 |
| 是否必填 | boolean | 当前版本默认必填 |
| 默认值 | any | 当前版本可不强调 |
| 说明 | string | 给使用者看的输入提示 |

补充规则：

- 开始节点只负责定义“使用该工作流，用户必须填写的内容”。
- 开始节点中的输入项由工作流创建者手动新增。
- 当前版本不在开始节点中配置主产物类型。
- 当前版本不在开始节点中配置最终产出数量。
- 输入项对用户优先展示友好字段名称，不默认暴露技术变量名；系统内部可自动生成稳定 key 用于后续引用。

#### 3.3.6 生成类节点规则 (文本/图像/视频)

工作流的核心生成能力由文本、图像、视频生成节点承载，所有生成节点遵循一致的组件规范与联动规则：

1. **模型与提示词**
   - **模型选择**：每类节点提供适配的模型下拉框（如 DeepSeek V4、Banana 2 等）。
   - **提示词划分**：文本生成拆分为“系统提示词”和“用户提示词”（必填）；图像/视频生成使用单一“生成提示词”（必填）。
   - **变量聚合引入**：每个提示词框均配套原生折叠面板式的“引入变量”模块。变量严格按来源节点名称分组展示，且针对提示词输入**仅过滤展示文本类输出**（如 text、count、text.generate 产物），避免无效跨模态插入。
2. **参考文件与动态校验**
   - **模型驱动**：是否允许上传/选择参考文件，以及支持的文件类型（图片/视频），由当前选中的模型能力决定。
   - **支持模式**：支持手动上传文件（前端带有 `accept` 原生校验），也支持通过 `optgroup` 节点分组下拉框选择上游节点输出的中间多媒体产物。
   - **不支持反馈**：若切换至不支持参考文件的模型（如 Doubao Seed 2.0），区域自动隐藏上传组件并置灰提示不可用。
3. **输出规格配置**
   - 图像节点支持配置“比例/分辨率”（如 16:9, 1:1 等），并支持在选择“自定义”时额外展示文本框供用户手动输入（如 1920x1080）。
   - 视频节点支持配置“比例/分辨率”（含“自定义”选项）以及动态联动的“时长”选项。
4. **生成数量与变量自动组合遍历**
   - **基础行为**：默认允许用户手动填写本节点的“生成数量”（如 10条文案、4张图片）。
   - **遍历联动逻辑**：当本节点的提示词或参考文件中插入了上游的变量（即 `{{xxx}}` 标签）时，运行逻辑自动变更为**组合遍历输出**。
   - **UI反馈**：触发联动后，本节点的“生成数量”输入框将置灰禁用，系统自动计算上游对应节点的产出乘积，并展示如 `X (由上游变量组合决定)`。

#### 3.3.7 结束节点规则

- 结束节点定义最终输出类型。
- 一个工作流只能有一个主产物类型：文案、图像或视频。
- 工作流内部可以产生中间文案、图片、视频，但最终输出只能归到一个主产物类型下。
- 结束节点除主产物类型外，还必须单选一个“输出内容”，即从之前节点输出中选择一个作为最终结果。
- 结束节点中的“输出内容”候选项需按当前主产物类型过滤：文案只显示文本类上游输出，图片只显示图片类上游输出，视频只显示视频类上游输出。
- 结束节点中的“输出内容”候选项文案当前仅展示节点名称。
- 结束节点走完后，工作流运行任务状态变为已完成，并在我的任务中展示结果。

#### 3.3.8 人工审查节点规则

- 工作流中允许配置多个人工审查节点。
- 人工审查节点可以出现在文案生成后、图片生成后、视频合成前等不同阶段。
- 如果多个上游节点连到同一个人工审查节点，代表这些上游产物需要一起被审查。
- 同一个人工审查节点必须等所需上游内容都准备好后，才进入可审查状态。
- 审查通过的内容进入下游节点。
- 审查拒绝的内容不进入下游节点，后续可按配置决定是否补生成、跳过或终止。
- 审查节点不应把每个物料拆成独立任务频繁打扰用户，应在任务中心聚合展示。
- 当前版本的人工审查节点在编辑器中只保留“节点名称”配置，不额外开放审查模式等复杂策略项。

#### 3.3.9 测试与发布

| 状态 | 说明 |
|---|---|
| 未测试 | 节点或工作流尚未执行测试 |
| 测试中 | 正在运行测试 |
| 测试通过 | 测试输入下执行成功 |
| 测试失败 | 测试输入下执行失败，需要调整配置 |

发布规则：

- 编辑器应自动保存当前配置，并让用户感知最近一次保存状态。
- 关键节点测试通过后才允许发布。
- 开始节点、结束节点、主链路连通性必须校验通过。
- 存在孤立节点时允许保留当前编辑状态，但发布前必须提示。
- 发布后可被自己使用；公开后可进入团队或广场范围。

#### 3.3.10 操作按钮

| 按钮 | 说明 | 结果 |
|---|---|---|
| 编辑基础信息 | 编辑工作流名称、描述、封面、可见性权限 | 工作流基础信息同步更新 |
| 自动保存 | 修改节点配置后自动保存当前画布 | 用户可感知最近一次保存状态 |
| 试运行 | 按开始节点表单试运行整个工作流 | 展示运行输入、执行中状态与成功/失败结果 |
| 添加节点 | 从节点库新增节点 | 工作流画布新增节点 |
| 删除节点 | 删除当前节点 | 删除节点和关联连线；开始/结束节点不可删除 |
| 删除连线 | 删除当前连线 | 连线移除 |
| 关闭配置 | 关闭当前配置视图 | 返回工作流画布视图 |

### 3.4 我的任务页

文件：`AI 创作/我的任务.html`  
入口：左侧导航「我的任务」；工作流运行后自动进入或由通知跳转  
职责：集中查看 AI 创作任务，工作流任务按最终产物类型融合到文案、图像、视频任务流中。

#### 3.4.1 页面结构

| 区域 | 说明 |
|---|---|
| 类型 Tab | 文案 / 图像 / 视频，工作流任务按最终产物类型进入对应 Tab |
| 筛选区 | 按任务状态、来源、项目、时间等筛选 |
| 任务列表 | 展示普通生成任务和工作流任务 |
| 左侧悬浮通知 | 聚合生成中、审查中、未查看等任务入口 |
| 底部输入区 | 保留普通任务再生成或新生成入口 |

#### 3.4.2 工作流任务展示规则

工作流任务分两类展示：

| 工作流任务状态 | 展示方式 |
|---|---|
| 未完成 / 运行中 / 审查中 | 展示工作流运行态卡片 |
| 已完成 | 展示普通结果卡片 |

运行态卡片结构：

```text
来源标签 + 工作流名称

开始  →  当前节点（开始审查/生成中）  →  待审查（N）  →  结束

查看详情 / 重新编辑 / 停止生成                         状态 + 时间
```

设计要求：

- 卡片视觉必须和其他任务卡片统一，不允许使用脱离系统的灰底线框图风格。
- 工作流卡片右上角不显示任何标签（如产物类型、输出规格等）。
- 工作流卡片底部不展示「已产出 / 预计总数」等数量信息。
- 任务卡片必须保留「查看详情」操作，点击打开弹窗展示当前各节点的详细运行情况（按节点聚合展示成功/失败状态、任务参数及失败原因）。
- 老板手绘图仅作为信息结构参考，不作为视觉样式稿。
- 当前需要用户处理的审查节点应被高亮强调，并提供清晰、明确的进入审查处理入口。
- 如果当前没有可审查内容，则展示「生成中」或「生产中」，不展示可点击的审查按钮。
- 未运行到的审查任务可以在界面上简写为「待审查（N）」，但产品和接口语义上表示“后续还未运行到的审查节点/审查批次”，不是已经可操作的待办。
- 当前可操作审查和未运行到审查必须在状态上区分清楚，避免用户误以为所有审查都已经可以处理。
- 对用户的主表达应优先聚焦“当前可审查多少项、后续还有多少未运行到的审查任务、当前是否仍在继续生成”，而不是平铺展示所有中间节点复杂性。
- 审查相关快捷动作（如自动通过、全部通过）只能作为次级效率操作，不应抢占「开始审查」的主语义。
- 已完成工作流不再展示流程图，直接展示最终结果。

#### 3.4.3 文案任务卡片展示规则

文案任务需要同时表达两层信息：

1. 外层任务卡片代表一次生成请求，例如“帮我重写 20 条小红书文案”。
2. 内层文案结果行代表本次任务产出的每一条可被单独处理的文案。

因此，文案任务不能只在任务卡片中展示一个合并后的段落，也不能采用聊天消息气泡样式。应使用“任务卡片 + 文案结果行列表”的结构。

文案任务卡片结构：

```text
任务来源 + 生成要求摘要                         模型 + 项目/场景

01  文案内容摘要...                         已完成   复制 / 编辑 / 再生 / 采用
02  文案内容摘要...                         已完成   复制 / 编辑 / 再生 / 采用
03  文案内容摘要...                         已完成   复制 / 编辑 / 再生 / 采用

共计产出 20 条文案 · 每条文案可单独处理          查看全部

重新编辑 / 再次生成 / 查看全部                 状态 + 时间
```

文案结果行字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| 行序号 | number | 当前预览列表中的展示序号，从 1 开始 |
| 文案内容 | string | 单条文案正文，列表态单行截断，详情态可展开全文 |
| 单条状态 | enum | 生成中 / 已完成 / 已失败 / 已采用 / 已入库 |
| 操作按钮 | array | 复制、编辑、再生、采用等单条操作 |
| 来源任务 ID | string | 所属任务，用于回溯生成请求 |
| 文案结果 ID | string | 单条文案唯一标识，用于单条再生、采用、入库 |

状态展示规则：

| 任务状态 | 文案结果行展示 |
|---|---|
| 等待中 | 可展示 1-4 条等待占位行，提示任务排队中 |
| 生成中 | 已产出的文案展示为真实结果行；未产出的数量用骨架行或生成中占位展示 |
| 已完成 | 展示前 3-4 条真实文案结果行，底部展示总产出数量和查看全部入口 |
| 部分失败 | 成功文案正常展示；失败文案展示失败状态，可提供单条再生 |
| 已失败 | 若无任何产出，展示失败占位；若已有部分产出，则按部分失败处理 |
| 已中止 | 保留已产出的文案结果，未完成部分不再继续展示生成中 |

操作规则：

| 操作 | 作用范围 | 说明 |
|---|---|---|
| 复制 | 单条文案 | 将当前文案复制到剪贴板，不影响任务状态 |
| 编辑 | 单条文案 | 打开单条文案编辑入口，编辑结果应保留与原任务的关系 |
| 再生 | 单条文案 | 只重生成当前这一条，不重新生成整批文案 |
| 采用 | 单条文案 | 标记该文案被当前业务采用，可进入后续投放、素材或脚本链路 |
| 查看全部 | 整个任务 | 进入全部结果页，展示本任务所有文案 |
| 再次生成 | 整个任务 | 使用原任务输入重新发起一批文案生成 |
| 重新编辑 | 整个任务 | 回到生成输入表单，允许修改原始需求后重新提交 |

设计要求：

- 文案结果行应是紧凑横条，不做大卡片，避免一条文案占用过多垂直空间。
- 单条操作按钮必须跟随文案结果行展示，不能只放在任务卡片底部，否则用户无法判断操作对象。
- 预览列表默认展示 3-4 条；数量较多时通过「查看全部」进入完整结果。
- 文案内容在任务卡片内默认单行截断，移动端可放宽为 2 行截断。
- 状态标签必须使用有文字的标签，例如“已完成”“生成中”“失败”，不使用纯色圆点。
- 文案任务仍然属于“我的任务”统一任务卡片体系，外层卡片的底部状态、时间、重新编辑、再次生成等操作保持与其他任务一致。

#### 3.4.4 数据字段

| 字段 | 类型 | 说明 |
|---|---|---|
| 任务 ID | string | 工作流运行任务唯一标识 |
| 工作流 ID | string | 来源工作流定义 |
| 工作流名称 | string | 卡片标题 |
| 主产物类型 | enum | 文案 / 图像 / 视频 |
| 输出规格 | string | 如 1920×1080、9:16、文案条数等 |
| 所属项目 | string | 任务所属项目 |
| 当前状态 | enum | 等待中 / 生成中 / 审查中 / 已完成 / 已失败 / 已中止 |
| 当前节点 | string | 当前运行到的节点 |
| 可审查数量 | number | 当前需要用户处理的审查数量 |
| 未运行到审查数 | number | 后续还未运行到的审查节点数量 |
| 已完成审查数 | number | 已处理审查数量 |
| 创建时间 | datetime | 任务创建时间 |
| 完成时间 | datetime | 任务完成时间 |
| 输出结果 | WorkflowResult[] | 最终产物列表 |

#### 3.4.5 工作流任务状态流转

```text
等待中 → 生成中 → 审查中 → 生成中 → 已完成
等待中 → 生成中 → 已完成
等待中 / 生成中 / 审查中 → 已中止
等待中 / 生成中 / 审查中 → 已失败
已失败 → 重新编辑 / 重试 → 等待中
```

状态说明：

| 状态 | 说明 | 用户感知 |
|---|---|---|
| 等待中 | 任务已提交，尚未开始执行 | 可以查看详情、停止生成 |
| 生成中 | 自动节点正在运行 | 展示生产中，不要求用户处理 |
| 审查中 | 至少一个审查节点已准备好，需要用户处理 | 展示开始审查入口 |
| 已完成 | 结束节点已完成，最终结果已生成 | 展示结果，不展示流程图 |
| 已失败 | 工作流执行失败 | 展示失败状态和重试/查看详情 |
| 已中止 | 用户主动停止或系统中止 | 展示已中止状态 |

#### 3.4.6 操作按钮

| 按钮 | 说明 | 结果 |
|---|---|---|
| 开始审查 | 当前有可审查任务时展示 | 打开审查台 |
| 查看详情 | 查看工作流运行情况 | 弹出详情窗展示各节点成功/失败状态及任务参数 |
| 重新编辑 | 基于当前工作流进入编辑器 | 打开工作流编辑器 |
| 停止生成 | 停止当前运行任务 | 二次确认后状态变为已中止 |
| 查看全部 | 查看完整结果 | 普通任务打开全部结果页；工作流任务打开工作流全部结果页 |
| 重试 | 对失败任务重新发起 | 任务重新进入等待中 |
| 入库 | 将结果进入对应资产库 | 创建入库动作或任务 |
| 下载 | 下载结果 | 创建下载动作或直接下载 |

### 3.5 审查台

文件：可由 `AI 创作/我的任务.html?view=review&taskId=...` 承接，后续也可拆独立页  
入口：我的任务 → 工作流运行态卡片 → 开始审查  
职责：集中处理工作流人工审查节点产生的待审内容。

#### 3.5.1 页面结构

| 区域 | 说明 |
|---|---|
| 任务信息区 | 展示工作流名称、审查节点名称、任务状态 |
| 审查统计区 | 待审、已通过、已拒绝、未运行到等统计 |
| 内容审查区 | 展示当前待审内容 |
| 操作区 | 通过、拒绝、批量通过、提交等 |

#### 3.5.2 数据字段

| 字段 | 类型 | 说明 |
|---|---|---|
| 审查任务 ID | string | 审查任务唯一标识 |
| 工作流任务 ID | string | 所属 WorkflowRun |
| 审查节点 ID | string | 所属人工审查节点 |
| 审查节点名称 | string | 如“审查A”“文案初审”“图片终审” |
| 待审内容 | WorkflowResult[] | 当前需要审查的内容 |
| 已通过内容 | WorkflowResult[] | 已通过内容 |
| 已拒绝内容 | WorkflowResult[] | 已拒绝内容 |
| 未运行到数量 | number | 后续还未到达审查节点的数量 |
| 审查状态 | enum | 未开始 / 可审查 / 审查中 / 已完成 |

#### 3.5.3 审查状态流转

```text
未运行到 → 可审查 → 审查中 → 已完成
可审查 / 审查中 → 已跳过（仅特定配置允许）
```

内容级状态：

```text
待审 → 通过
待审 → 拒绝
拒绝 → 撤回到待审
通过 → 撤回到待审（仅下游未消费时允许）
```

#### 3.5.4 审查逻辑

- 审查台可以展示一个或多个同时可处理的审查任务，但默认聚焦当前触发入口对应的审查任务。
- 如果同一工作流中存在多个并行审查点同时 ready，可以在审查台中切换处理。
- 如果审查节点之间存在前后依赖，下游审查不能修改上游已被消费的审查结果。
- 已通过 / 已拒绝结果仅在“下游尚未消费”时允许撤回到待审；一旦下游已经基于该结果继续处理，前序审查结果不可再修改。
- 用户完成当前审查后，系统根据工作流配置继续执行下游节点。
- 系统通知或左侧悬浮提醒应按“审查节点 ready”进行聚合提醒，不按单个产物逐条提醒。
- 若一个工作流需要生产 50 个产物，不应给用户制造 50 个独立审查通知；用户更需要一次进入审查台后集中处理。
- 审查不是聊天，不使用聊天窗口样式。
- 文案审查内容使用垂直列表展示。
- 图片和视频审查可使用卡片或网格，但操作必须清晰可见。
- 审查页顶部或摘要区应明确区分至少三类信息：当前可审查数量、后续未运行到的审查任务数量、仍在继续生成/运行的数量。

#### 3.5.5 操作按钮

| 按钮 | 说明 | 结果 |
|---|---|---|
| 通过 | 单项通过 | 当前内容进入通过集合 |
| 拒绝 | 单项拒绝 | 当前内容进入拒绝集合 |
| 批量通过 | 批量通过选中项 | 选中项进入通过集合 |
| 批量拒绝 | 批量拒绝选中项 | 选中项进入拒绝集合 |
| 撤回 | 撤回已处理结果 | 返回待审，受下游消费状态限制 |
| 提交审查 | 完成当前审查节点处理 | 推动工作流进入下游 |
| 返回任务 | 返回我的任务 | 回到任务列表 |

### 3.6 全部结果页

文件：`AI 创作/全部结果-统一版.html`（普通任务）、`AI 创作/工作流全部结果页.html`（工作流任务）
入口：我的任务 → 查看全部
职责：展示任务完整产物，并承接下载、入库、预览和待审核任务的后续处理。

#### 3.6.1 页面原则

- 任务结果页应根据任务类型自动导向：普通任务导向统一结果页，工作流任务导向工作流专用结果页。
- 结果页应尽量复用系统的整体框架与阅读习惯。
- 工作流专用结果页需承接文案、图像、视频三类最终产物，并统一排版。
- 结果页需能展示工作流不同运行阶段产出的最终结果。- 本页需要统一承接两类来源：
  - 通用生成任务
  - 工作流任务
- 本页需要统一承接三类最终产物：
  - 文案
  - 图像
  - 视频
- 本页需要统一承接五类展示状态：
  - 排队中
  - 生成中
  - 已完成
  - 已中止
  - 待审核

#### 3.6.2 展示规则

| 产物类型 | 展示方式 |
|---|---|
| 文案 | 垂直列表，一行一条 |
| 图像 | 沿用旧结果页风格的缩略图结果区 |
| 视频 | 沿用旧结果页风格的缩略图结果区，并提供播放预览入口 |
| 混合中间产物 | 不作为最终结果页主展示，需在任务详情或审查台中查看 |

补充规则：

- 结果页不再提供生成模型（如 Seedance 2.0 等）的筛选能力。
- 图像与视频结果页不再提供生成比例（如 9:16、16:9、1:1 等）的分类筛选标签，统一在一个网格视图中混排展示。
- 工作流结果页头部应展示任务名和工作流名。其中，主标题需展示“任务名称AAAA”（代指具体任务名称），而不再拼接冗长的配置信息（如项目名、视频配置等）。
- 结果页头部不再平铺展示具体的提示词文本以及参考图信息，这部分复杂输入由“查看详情”弹窗承接。
- 文案结果页的主要变化应集中在结果展示区本身，不需要为文案结果单独重做整页布局。
- 图像 / 视频结果区的展示样式、按钮相对位置、密度切换位置，应尽量与旧结果页保持一致。

#### 3.6.3 状态口径

| 页面状态 | 说明 |
|---|---|
| 排队中 | 任务已提交，尚未开始产出；结果区主要展示等待占位 |
| 生成中 | 已有部分结果产出，未完成部分继续展示 loading / pending 占位 |
| 已完成 | 任务当前已结束展示；通用生成原先的“已成功 / 已失败 / 部分失败”在统一结果页展示口径中统一归入“已完成” |
| 已中止 | 用户主动停止任务，保留已产出结果，不再继续生成 |
| 待审核 | 工作流已运行到人工审核节点；保留已产出结果，并提供明确的去审核入口 |

> 说明：旧 `全部结果_部分失败.html`、`全部结果_全部失败.html` 仍保留为历史原型文件，但当前统一结果页不再把它们作为主入口状态页口径。

#### 3.6.4 操作规则

| 按钮 | 说明 | 结果 |
|---|---|---|
| 查看详情 | 查看工作流输入或任务参数详情 | 打开详情弹窗 |
| 去审核 | 工作流待审核状态下进入审核节点 | 打开审查台或对应审核入口 |
| 大 / 中 / 小 | 调整图像 / 视频结果区缩略图密度 | 仅影响结果展示区 |
| 批量选择 | 选择多个图像 / 视频结果 | 进入批量操作模式 |
| 下载全部 | 下载当前结果集合；文案场景可表现为复制全部 | 创建下载动作或复制动作 |
| 全部入库 | 将当前结果集合入库 | 进入素材 / 物料 / 原料链路 |
| 中止生成 | 停止当前仍在进行中的任务 | 页面状态切换为已中止 |
| 下载 | 下载单个图像 / 视频结果 | 下载文件 |
| 预览 | 查看大图或播放视频 | 打开预览层 |
| 返回任务 | 返回任务卡片 | 回到我的任务 |

## 四、工作流状态总览

### 4.1 工作流定义状态

```text
草稿 → 已发布 → 已下架
已下架 → 已发布
草稿 → 已删除
已下架 → 已删除
```

| 状态 | 说明 | 可操作 |
|---|---|---|
| 草稿 | 未发布，只创建者可编辑和测试 | 编辑、测试、发布、删除 |
| 已发布 | 可被使用，按权限可见 | 使用、编辑、下架、收藏 |
| 已下架 | 不再允许新使用，历史任务不受影响 | 编辑、重新发布、删除 |
| 已删除 | 逻辑删除 | 不展示 |

### 4.2 工作流运行任务状态

```text
等待中 → 生成中 → 审查中 → 生成中 → 已完成
生成中 → 已完成
等待中 / 生成中 / 审查中 → 已失败
等待中 / 生成中 / 审查中 → 已中止
已失败 → 重试 → 等待中
```

### 4.3 节点运行状态

| 状态 | 说明 |
|---|---|
| 未开始 | 节点尚未被触发 |
| 等待上游 | 上游节点未完成 |
| 运行中 | 当前节点正在执行 |
| 待审查 | 人工审查节点已准备好 |
| 已通过 | 审查节点完成且有通过结果 |
| 已完成 | 节点执行完成 |
| 已失败 | 节点执行失败 |
| 已跳过 | 因条件或配置跳过 |

### 4.4 审查任务状态

| 状态 | 说明 |
|---|---|
| 未运行到 | 工作流还没有执行到该审查节点 |
| 可审查 | 审查节点所需内容已准备好 |
| 审查中 | 用户正在处理或已有部分处理 |
| 已完成 | 审查节点处理完成 |
| 已跳过 | 特定配置允许跳过 |

### 4.5 结果状态

| 状态 | 说明 |
|---|---|
| 生成中 | 结果仍在生成 |
| 待审 | 结果进入审查节点，等待人工处理 |
| 已通过 | 审查通过，可进入下游 |
| 已拒绝 | 审查拒绝，不进入下游 |
| 已完成 | 最终产物完成 |

## 六、质量控制与同质化

### 6.1 同质化检测原则

- 工作流生成的最终产物在入库前必须支持同质化检测。
- 若生成物料与库内已有素材重复率大于 80%，系统需标记为“同质化过高”。
- 同质化过高的产物不能自动入库，必须进入人工确认或待操作流程。
- 同质化检测不阻塞工作流编辑器本身，只影响运行结果入库链路。

### 6.2 结果处理

| 检测结果 | 处理方式 |
|---|---|
| 合格 | 允许进入后续入库或使用流程 |
| 同质化过高 | 标记风险，要求人工确认 |
| 检测失败 | 展示检测失败，可重试或人工处理 |

## 七、文案、图片、视频展示规则

### 7.1 文案结果

- 文案结果必须使用垂直列表。
- 每条文案单独成行，支持复制、编辑、单条再生、采用、通过/拒绝等操作。
- 禁止使用图片式网格卡片展示文案。
- 在我的任务中，文本生成任务必须是标准任务流列表样式，禁止使用聊天窗口样式。
- 如果一次任务生成多条文案，任务卡片内必须展示多个独立文案横条，而不是把多条文案合并成一段编号文本。
- 单条文案横条必须明确操作对象，按钮与文案同行展示；任务级操作如重新编辑、再次生成、查看全部仍放在卡片底部。

### 7.2 图片结果

- 图片结果可使用缩略图网格。
- 缩略图应支持预览、下载、入库、审查操作。
- 生成中可以使用统一 loading 占位。

### 7.3 视频结果

- 视频结果展示缩略图、时长、播放入口。
- 视频预览应支持播放，不只展示静态图。
- 视频处理中的音轨应作为独立对象在数据结构中保留。

### 8.2 原型承载约束

- 当前工作流相关页面以可独立预览的静态原型形式推进，用于承载页面结构、交互状态和信息展示。
- 页面需继续遵循系统既有的统一承载方式，保持菜单、页面打开方式和跨页跳转体验一致。
- 工作流编辑器、我的任务、审查台、结果页等关键页面应优先保证稳定可预览、状态可演示、路径可走通，而不是引入额外使用门槛。

### 8.3 统一设计规范

| 规范项 | 要求 |
|---|---|
| 页面背景 | 保持系统统一的浅色工作台背景 |
| 页面间距 | 页面内外留白保持紧凑一致，避免出现过大的空白断层 |
| 页面承载关系 | 页面作为系统内部业务页存在，不单独承担导航框架职责 |
| 视觉一致性 | 卡片、筛选区、操作区、空状态等需与系统现有页面保持一致的视觉语言 |
| 工作流任务卡 | 视觉上与普通任务卡一致，只在内容结构上展示流程差异 |
| 文案展示 | 垂直列表，一行一条 |
| 广场定位 | 独立页面，不内嵌在通用生成页 |

| 能力 | 说明 |

## 十、验收标准

### 10.1 通用生成页验收

- 顶部文案/图像/视频切换能同步影响工作流快捷区。
- 工作流快捷区只展示收藏和我的，不展示完整广场。
- 有明确入口跳转到工作流广场。
- 新建工作流弹窗字段完整，封面不是第一强制项。

### 10.2 工作流广场验收

- 菜单中存在工作流广场入口。
- 页面可搜索、筛选、收藏、使用工作流。
- 收藏后的工作流能在通用生成页收藏区体现。

### 10.3 工作流编辑器验收

- 可以从左侧节点库拖拽节点到画布。
- 可以连线、删除连线、删除普通节点。
- 开始节点和结束节点初始化即存在，且不可删除。
- 可以配置节点参数。
- 可以定义开始节点输入。
- 可以在结束节点中定义主产物类型，并单选一个按类型过滤后的上游输出作为最终结果。
- 工作流基础信息支持修改名称、描述、封面和权限。
- 用户能够感知自动保存状态。
- 试运行能够完成输入、执行中、成功/失败结果展示。

### 10.4 我的任务验收

- 工作流任务按最终产物类型进入文案、图像或视频 Tab。
- 运行中工作流展示流程状态卡。
- 流程状态卡视觉风格与普通任务卡一致。
- 可审查时展示开始审查入口，且其语义优先级高于自动通过、全部通过等快捷操作。
- 运行中卡片需明确区分“当前可审查数量”和“后续未运行到的审查任务数量”，避免两者混淆。
- 完成后展示最终结果，不再展示流程图。
- 文案任务卡片内的多条文案必须拆成独立横条展示。
- 每条文案横条必须有清晰的单条操作入口，如复制、编辑、再生、采用。
- 文案任务生成中状态必须同时表达“已产出数量”和“剩余仍在生成”，不能只展示空白或单一句“生成中”。

### 10.5 审查台验收

- 可展示当前可审查、已通过、已拒绝、未运行到等统计信息。
- 可单项通过/拒绝。
- 可批量通过/拒绝。
- 已处理结果仅在下游未消费时允许撤回。
- 提交后工作流能继续下游。
