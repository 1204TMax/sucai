## 一、 核心业务逻辑与流程约束

1. **宏观定位：半自动化的流水线（Check Point 机制）**
   - 工作流**不是**完全黑盒的端到端黑魔法，而是一个带有“人工审查卡点”的半自动流水线。
   - 每当工作流运行到“人工审查 (Review)”节点时，系统必须暂停当前分支的自动流转。
   - 审查逻辑不是“全量完成后再统一看”（Batch），而是倾向于“边生产边审核”（Streaming），以提高并发效率。

2. **输入约束与目标数量管理**
   - 用户在启动（新建/执行）一个工作流时，第一步（START节点）必须：
     - a) 明确输入一组或多组“原料（物料池）”。
     - b) 明确设定本次生产的“目标产出数量”（例如：我要裂变生成 100 套素材）。
   - 这里的 START 节点不要只理解成“给工作流起个名字”。在界面与数据结构上，都要把“原料来源”和“目标数量”当成正式配置项，而不是后面再补的附属信息。
   - 工作流的后续执行（循环次数、质量校验）将严重依赖这个“目标数量”。如果不符合要求，算法/人工将提前介入提示“原料不足以生成X套”。

3. **数据结构与流转：结构化的“素材包”**
   - 节点之间传递的不是单纯的字符串或二进制文件，而是特定类型的“数据结构体”。
   - 例如：视频拆解节点（拆解 Processor）的输出，不是几段 mp4，而是一个包含：`[音轨对象, 片段A结构, 片段B结构, 字幕文本, 原视频关联ID]` 的复合结构体。
   - 下游节点（例如：文案衍生、自动混剪）必须能识别并解析这种特定的结构体。

## 二、 核心功能模块 (Processor) 规划

*注：虽然长期目标非常庞大（换脸、去水印、智能抠图等），但 v1.0 必须聚焦于最基础的“跑通框架”。*

1. **v1.0 必须包含的基础 Processor**
   - **基础生成类**：文本处理 Processor、图像处理 Processor、视频处理 Processor。
   - **业务强相关类**：
     - **素材拆解 Processor**：输入爆款视频，拆解出音轨、结构段落等，输出“拆解结构体”。
     - **素材衍生 Processor**：基于拆解出的结构体（如音轨、特定画面），进行 1转10 的裂变。
     - **自动混剪 Processor**：接收人工审查过的“衍生合格素材”，按照预设的（或原视频的）逻辑顺序进行无脑拼接组合，输出最终成片。
   - **控制类**：人工审查卡点 Processor（拦截、展示内容、支持黑白名单勾选通过）。

2. **功能降级与内聚原则**
   - 像“图片去水印”、“图片等比例裁剪”、“视频超分”这种**通用且高频**的前置/后置动作，**不应该**作为独立的 Processor 暴露给用户去连接。
   - 它们应该被内聚（封装）在“图片生成 Processor”或“视频生成 Processor”的内部配置项中（例如作为一个 Checkbox：“生成后自动去水印并裁剪至 9:16”）。
   - 换句话说，工作流编辑器里对用户暴露的，应该优先是“业务动作”或“明确处理阶段”，而不是一堆很碎的技术步骤；否则节点一多，工作流会迅速变成工程师玩具，而不是业务同学可用的生产工具。

## 三、 用户界面 (UI/UX) 设计规范

1. **工作流入口与广场 (通用生成页)**
   - 页面需要明确的层级划分，避免大量工作流堆砌导致用户迷失。
   - **最高优（置顶）**：“我的收藏”工作流。
   - **次高优**：“我创建的”工作流（支持编辑、删除）。
   - **最低优（或采用侧边栏/独立Tab划分）**：“工作流广场”（全公司公开的模板，只能查看、使用和收藏，不能编辑）。

2. **任务中心聚合 (我的任务)**
   - **坚决摒弃**目前满屏刷新的单条任务列表模式（因为自动工作流一跑就是成百上千条记录，会瞬间淹没有效信息）。
   - **聚合展示**：任务必须以“工作流执行批次”为维度进行聚合（例如：“4月24日 裂变任务包 (共100条)”）。
   - **大区块导航**：页面顶部必须有极具视觉冲击力、空间占比大的“宏观导航看板”，直观展示：
     - `正在进行中的工作流` (带进度条/动画)
     - `待人工审查的节点` (最重要，标红提示用户需要去操作了)
     - `已完成/失败`
   - 状态联动：列表中不再仅仅是“生成中/已完成”，必须增加 `待审查` 状态。点击“待审查”后，进入专门的全屏核对页面（支持打钩、踢出黑名单，然后一键放行至下一节点）。
   - 这里需要额外强调：`任务类型`、`结果类型`、`任务状态` 是三个不同维度，后续实现时不要再混在一起。
     - 例如：`工作流` 是任务类型，不是结果类型；
     - `图像 / 视频 / 文案` 是结果类型，不是任务状态；
     - `待审查 / 生成中 / 已完成 / 已失败` 才是状态。
   - 工作流任务卡不要简单套用普通结果卡的思路。普通任务卡的主角通常是“图片/视频结果阵列”，但工作流任务卡在中间态（生成中、待审中）时，主角应该是“当前 processor / 当前 checkpoint / 待处理数量 / 整体进度”。
   - 只有当工作流已经走到最终完成态时，卡片中间区域才适合退回到现有那种“查看结果缩略图 + 查看全部”的表现。
   - 为了减少认知负担，工作流卡在列表中要尽量沿用现有任务卡的外框与区域结构（标题区、主体区、底部操作区、状态区），不要另起一套完全陌生的视觉体系；但是在中间态时，主体区不能再让“图片阵列”当主角，而应该让“流程状态”当主角。
   - 如果当前工作流已经产出了一批待审核内容，那么卡片上的最高优先操作应该是“进入审查 / 去审查”，优先级高于“查看进度”。

## 四、 补充说明：原料库的定位
- **原料**：拆解出来的音轨、尚未完成加工的片段等，属于“原料”。它们专门服务于 AI 创作系统。
- **物料**：最终可以直接用于投放合成的成品，才进入“物料库”。
- 两者在物理存储上可以是同一个文件（通过指针关联），但在业务界面上必须隔离。原料库主要用于追溯“这个半成品是哪个工作流的哪个任务产出的”。

## 五、 核心技术与 UI/UX 实现规范 (来自明确指令的强制约束)

为避免开发过程中的信息衰减，以下原则属于**必须遵守的强制约束**（不可随意发挥）：

1. **版本控制与文档管理 (SSOT 原则)**：
   - **分支要求**：所有开发和提交必须严格在 `工作流` 分支上进行。
   - **文档唯一性**：文档维护采用 Single Source of Truth (SSOT) 模式。必须直接更新 `/Users/dashan/Documents/dev/sucai-main/文档/未命名文件夹` 目录下的源文档，**坚决禁止**创建类似 `_v2`、`_final` 这样带有版本号的新副本。

2. **工作流编辑器交互**：必须实现“拖拽式、画板式”的节点编辑交互（由于内部网络与跨域问题，现阶段必须采用纯原生 HTML/JS/CSS 实现，保证零依赖与 0 白屏概率）。
3. **菜单结构排布**：新增加的“原料库”页面，必须精确地放在左侧菜单的『AI 创作』分组下，且必须紧挨在『项目管理』的上方。
4. **原料库管理维度**：明确划分为“视频”、“图片”、“文案”三大模块，必须展示明确的产物溯源链路（标识该原料是手动上传还是由哪个 WF-ID / TSK-ID 产出）。
5. **文案工具输入配置**：文案生成能力必须支持配置『参考图片』(Reference Image)上传、『裂变数量』(Fission Count，1-100条) 设定，以及『文案风格』设置。
6. **文案结果呈现**：文案的生成产物在任何页面（包括结果页、任务预览）都**必须**以“垂直列表（一行一条）”的形式展现，坚决禁止套用图片视频的网格卡片（Grid）布局；另外，在“我的任务”中，文本生成必须是标准任务流列表样式，**严禁**使用聊天窗口（Chat）样式。
7. **审查逻辑 (Streaming Review)**：必须实现“流式审查”机制。即：当批处理任务到达人工审查节点时，节点本身是阻塞的，但**单个物料只要被点“通过”，就必须立刻流转到下游节点进行下一道工序**，绝对不要等这 100 个全审完才往下走。
8. **全局样式与路由**：页面 `body` 背景色必须严格限定为 `#f5f7fa`，padding 为 `8px`，坚决避免使用可能导致滚动条锁死的 `height: 100vh`。页面间的跳转和开Tab交互，统一使用 `parent.postMessage({action: 'openTab', ...}, '*')`。
9. **动态配置与连线机制**：工作流配置采用“动态表单”收集开始节点定义的输入参数，参数传递必须支持基于 `{{NodeID.Variable}}` 的上下文映射。
10. **节点抽象方式**：编辑器中的节点，不要长期停留在“示例节点枚举”的层级，而要尽快收口成 `processor` 类型体系。
    - 对用户暴露的是可理解的 Processor 分类（如输入、解析、生成、审查、输出）。
    - 节点库中应支持分类导航，方便未来扩展更多 processor，而不是节点一多就平铺堆满。
11. **END 节点语义**：结束节点默认表示“进入任务系统”，不要再把 END 节点设计成复杂的出库/回调路由配置器。
    - 也就是说，工作流跑完后，结果天然回到“我的任务”体系中继续被查看、审查、入库、追溯。
    - 不需要为了 END 再额外暴露一堆让用户难以理解的输出目标配置。
12. **最终产物追溯 (Traceability)**：所有合成与生成的最终资产必须附带元数据，强制链接回其对应的 `workflow_id` 与 `task_id`，支持无缝反向溯源。
13. **工作流编辑器首批节点收敛口径 (v1.0)**：当前编辑器首批只保留 6 类节点：`开始`、`文本生成`、`图片生成`、`视频生成`、`人工审查`、`结束`。
    - **开始节点**：名称固定为「开始」。配置项只包含：用户运行时输入表单（可新增，当前先支持「文件上传」「提示词/文本」两类）、目标产出类型（文案/图片/视频）、最终产出数量（1-100）、图片或视频的比例/分辨率。
    - **文本生成节点**：配置项为名称、模型、提示词、参考文件、生成数量。其中提示词与参考文件都必须支持三种来源：`自己填写/上传`、`继承开始节点输入`、`继承其他节点输出`。
    - **图片生成节点**：配置项为名称、模型、提示词、参考文件、生成数量、分辨率、是否去水印。其中提示词与参考文件同样支持三种来源：`自己填写/上传`、`继承开始节点输入`、`继承其他节点输出`。
    - **视频生成节点**：配置项为名称、模型、提示词、参考文件、生成数量、分辨率、是否去水印。其中提示词与参考文件同样支持三种来源：`自己填写/上传`、`继承开始节点输入`、`继承其他节点输出`。
    - **人工审查节点**：继续保留为审查卡点节点，核心配置聚焦在审查对象、审查标准、审查方式（逐条通过/拒绝或黑白名单勾选）以及是否允许“全部通过”。
    - **结束节点**：名称固定为「结束」，保持“进入任务系统”的单一语义，不再承载模型、提示词、参考文件、数量、分辨率等生成配置。
    - **Why**：这是对 1.0 范围的主动收敛。先把基础生产链路和参数传递机制跑通，再逐步扩展拆解、衍生、混剪等更复杂的 processor，避免一开始把编辑器做成无法落地的“工程师玩具”。
